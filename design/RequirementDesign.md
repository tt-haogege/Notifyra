# RequirementDesign

## 项目名称

**Notifyra**

## 1. 项目定位

Notifyra 是一个给个人或独立用户使用的聚合通知平台，聚合多种通知方式于一身，并支持通过平台配置或自然语言交互创建通知。

V1 聚焦于：
- 多账号但非团队协作
- 通知统一管理与推送
- 多轮 AI 辅助创建通知
- Webhook 触发通知
- 测试发送与接入示例生成

暂不纳入 V1：
- 团队能力
- 邮件转发
- NAS 日志接入
- 通用模板系统

这些能力作为后续模块演进。

---

## 2. 用户与范围

### 2.1 用户范围
- 支持多账号
- 支持注册
- 注册后可直接使用，无需审核
- 当前不考虑团队协作能力

### 2.2 账号规则
- 用户名唯一
- 用户名至少 5 位
- 密码至少 6 位
- 登录方式为账号密码登录
- 登录后通过 Token 进行接口鉴权
- 每个账号只能看到并操作自己的通知、渠道、推送记录、Webhook token、AI 配置和个人设置

---

## 3. V1 核心目标

### 3.1 聚合通知渠道
V1 首批支持以下渠道：
- 企业微信（Webhook）
- 飞书（Webhook）
- 钉钉（Webhook）
- Bark
- 通用 Webhook
- PushPlus

### 3.2 通知创建方式
V1 支持以下创建方式：
- 手动创建通知
- 通过 AI 多轮对话创建通知

### 3.3 通知触发类型
V1 支持以下通知触发类型：
- 一次性通知
- 周期性通知
- Webhook 触发通知

说明：
- “创建方式”用于描述这条通知是如何被创建出来的
- “触发类型”用于描述这条通知在什么条件下被执行
- 外部系统调用 Webhook 不属于“创建通知”，而是“触发已有的 Webhook 通知”

不纳入 V1：
- NAS 日志通知（后续独立模块）

---

## 4. 通知对象定义

产品侧统一使用“通知”作为核心对象名称，不使用“任务”作为用户面向概念。

一条通知至少包含以下字段：
- 通知名称
- 通知触发类型
- 发送渠道（可多选）
- 标题
- 正文
- 触发时间（按触发类型动态变化）
- 状态（默认启用）
- 下一次触发时间（启用状态下展示）
- 创建方式（系统内部可识别）
- 备注（可选，不作为创建页必填展示项）

字段定义：
- 通知名称：平台内部用于管理和识别该通知的名称
- 标题：实际推送到目标渠道中的消息标题

内容限制：
- 标题最多 100 字
- 正文最多 500 字

---

## 5. 通知创建规则

### 5.1 手动创建
手动创建页展示字段：
- 通知名称
- 通知触发类型
- 发送渠道
- 标题
- 正文
- 触发时间

触发时间规则：
- 当触发类型为一次性通知时，展示日期时间选择器
- 当触发类型为周期性通知时，展示 Cron 表达式输入框
- 当触发类型为 Webhook 通知时，不展示“触发时间”字段

创建页不展示：
- 状态（默认启用）
- 备注

### 5.2 AI 多轮创建
AI 创建通知采用多轮对话模式，而不是单轮猜测后直接落库。

规则如下：
- AI 与用户持续对话，直到拿到足够参数后创建通知
- 创建前最少需要以下信息：
  - 通知名称
  - 通知触发类型
  - 发送渠道
  - 标题
  - 正文
  - 触发时间或 Webhook 触发方式信息
- 某些字段允许 AI 自动补全，例如标题
- 参数足够后直接创建，不再额外弹确认卡片
- 创建成功后展示：
  - 创建成功提示
  - 结构化结果
  - 跳转到通知详情的入口

### 5.3 未配置 AI 时的行为
- 用户允许进入 AI 创建入口
- 当真正开始与 AI 对话时，如果未配置 AI，则提示用户去个人设置完成 AI 配置

## 6. AI 配置规则

AI 为用户自行配置，不采用平台统一配置。

配置位置：
- 个人设置

配置项包括：
- Base URL
- API Key
- 模型名称

### 6.1 时间偏好
AI 对自然语言时间的处理支持用户级偏好配置。

例如：
- 下午默认几点
- 晚上默认几点
- 明早默认几点

配置位置：
- 个人设置

---

## 7. 渠道管理

### 7.1 渠道配置方式
采用全局渠道配置方式：
- 用户先创建渠道
- 通知中直接选择已保存渠道

### 7.2 渠道管理支持操作
- 新增渠道
- 编辑渠道
- 删除渠道
- 启用 / 停用渠道
- 配置失败重试次数
- 查看并复制渠道调用信息

### 7.3 重试规则
- 重试配置在渠道层，不在通知层
- 用户可配置失败重试次数
- 最少 0 次
- 最多 3 次
- 默认 0 次
- 0 次表示不重试

### 7.4 渠道列表展示字段
- 渠道名称
- 渠道类型
- 状态
- 失败重试次数
- 最近一次使用时间
- 关联的通知

“关联通知”展示方式：
- 数量 + 名称结合
- 支持点击跳转到对应通知

### 7.5 渠道调用能力
当创建渠道后，系统需要为该渠道生成专属调用 token。

规则如下：
- 每个渠道拥有自己的渠道调用 token
- 平台提供固定调用路径
- 外部系统可通过固定路径 + 渠道 token 直接调用当前渠道
- 行为上类似 Webhook 通知的触发方式，但目标是直接命中渠道能力
- 渠道 token 需要支持复制与展示
- 渠道 token 应支持后续重置能力
- 完整 token 仅在生成或重置当下明文返回一次

### 7.6 删除与停用规则
#### 删除渠道
- 若渠道已被通知使用，则不允许删除
- 需先解绑相关通知

#### 停用渠道
- 若通知绑定多个渠道，停用其中一个渠道时，不自动停用整条通知
- 仅当通知已没有任何可用渠道时，通知才自动停止
- 自动停止时：
  - 通知状态需明确标记原因，例如“已停用（无可用渠道）”
  - 同时保留原因日志 / 操作记录
- 若通知没有可用渠道，则不允许重新启用，需先恢复或重新绑定可用渠道

## 8. 通知列表与详情

### 8.1 通知列表展示字段
- 通知名称
- 通知触发类型
- 渠道信息
- 状态
- 下一次触发时间
- 最近一次推送结果

渠道展示规则：
- 展示用户自定义渠道名 + 渠道类型
- 例如：工作群（飞书）
- 若绑定多个渠道：
  - 最多展示前 2 个
  - 其余折叠显示，例如：工作群（飞书）、主手机（Bark） +1

### 8.2 通知详情页区块
- 基础信息
- 渠道信息
- 内容信息
- 规则信息
- 最近推送记录（最近 10 条）

### 8.3 通知操作
V1 支持以下操作：
- 查看通知列表
- 查看通知详情
- 编辑通知
- 启用 / 停用通知
- 删除通知
- 查看推送记录

不纳入 V1：
- 立即发送一次

“测试发送”统一收口到测试模块。

补充说明：
- 测试模块发送的是测试消息，不等同于执行某条已存在通知一次
- 测试模块主要用于验证渠道可达性、消息展示效果和接入示例生成

---

## 9. 推送记录

### 9.1 推送记录字段
- 推送时间
- 通知名称
- 发送渠道
- 发送内容摘要
- 推送结果（成功 / 失败 / 部分成功）
- 失败原因
- 触发来源

### 9.2 多渠道发送结果规则
若一条通知发送到多个渠道，且结果不一致：
- 整体状态标记为“部分成功”
- 同时保留每个渠道的单独结果明细

### 9.3 推送记录展示规则
- 默认只展示最近 10 条推送记录

### 9.4 推送记录筛选
V1 支持以下筛选：
- 按结果筛选
- 按通知筛选
- 按时间范围筛选

### 9.5 记录边界说明
- 推送记录用于记录通知实际执行和发送结果
- 因渠道停用、无可用渠道等导致的状态变化，V1 不单独建设完整操作日志系统
- 此类信息以通知状态原因说明和必要的关联记录方式保留

---

## 10. Webhook 触发通知

### 10.1 Webhook 触发方式
当通知触发类型为 Webhook 时：
- 平台为每条通知生成固定调用路径与专属 token
- token 用于直接定位当前具体通知
- 外部系统通过固定路径 + token 直接命中当前通知
- 不通过系统统一 Webhook 后再二次手工分发

### 10.2 Webhook Token 规则
- 每条 Webhook 通知拥有专属 token
- 支持手动重置 token
- token 重置后旧 token 失效
- token 展示时仅展示头尾，中间脱敏
- 支持复制完整 token
- 完整 token 仅在生成或重置当下明文返回一次

说明：
- Webhook 通知 token 用于直接命中某条通知
- 渠道调用 token 用于直接命中某个渠道
- 两者能力类似，但作用对象不同，不能混用

### 10.3 Webhook 内容模板
Webhook 通知支持模板模式，仅用于 Webhook 类型通知。

支持范围：
- 正文支持模板变量
- 标题在 V1 中不支持模板变量，仍按普通标题处理
- 用户手动书写变量路径
- 仅支持请求体 body JSON 字段
- 例如：
  - {{body.message}}
  - {{body.device.name}}

暂不支持：
- Query 参数变量
- 请求头变量
- 自动识别并点选字段（后续体验优化项）

### 10.4 字段缺失策略
如果模板中的变量不存在：
- 保留原始占位符
- 例如 {{body.message}} 不存在时，仍展示 {{body.message}}

### 10.5 原始请求记录
Webhook 接收到请求后，需要保存原始请求内容，并在推送详情中展示。

V1 展示范围：
- 请求体 body
- 请求时间
- 来源 IP

暂不展示：
- 请求头
- query
- method 等完整请求信息

## 11. 首页 / 概览页

首页为概览型控制台，展示内容包括：
- 通知数量
- 成功数量
- 失败数量
- 最近的通知推送记录
- 快捷按钮

统计口径说明：
- 通知数量：当前用户名下的通知总数
- 成功数量：当前用户最近一段统计周期内的成功推送次数
- 失败数量：当前用户最近一段统计周期内的失败推送次数
- 最近的通知推送记录：按时间倒序展示当前用户最近发生的推送记录
- 具体统计周期在后续产品方案与技术方案中统一定义，V1 需保持前后端口径一致

快捷按钮包括：
1. 添加渠道
2. 新建通知
3. 测试模块
4. 推送记录

新建通知页面采用双入口：
- 上方：一句话创建
- 下方：手动创建入口

## 12. 测试模块

测试模块为独立模块，不归属渠道管理页。

### 12.1 目标
- 验证通知或渠道是否工作正常
- 生成对应接入示例代码

### 12.2 页面结构
测试模块固定采用顶部双入口结构：
- 测试通知
- 测试渠道

### 12.3 测试通知
- 先选择一条已有通知
- 系统读取该通知当前配置
- 执行一次测试发送
- 展示测试结果
- 同步生成对应通知调用示例代码

说明：
- 用于验证“这条通知当前配置是否可工作”
- 测试行为不等同于正式通知执行记录

### 12.4 测试渠道
- 先选择一个已有渠道
- 填写测试标题和正文
- 对该渠道发送测试消息
- 展示测试结果
- 同步生成对应渠道调用示例代码

说明：
- 用于单独验证渠道连通性和发送能力
- 不依赖某条通知存在

### 12.5 示例代码类型
V1 支持生成：
- curl
- JavaScript / Node.js
- Python

### 12.6 补充规则
- 测试模块不采用单入口随意发送方式
- 必须先明确测试对象是“通知”还是“渠道”
- 测试消息不等同于执行已有通知一次
- 测试模块是独立验证与接入辅助工具

---

## 13. 个人设置

个人设置页包含：
- 修改密码
- 更换头像
- 时间偏好
- OpenAPI 配置（AI 配置）

### 13.1 Token 定义
V1 中涉及的 token 需要明确区分：
- 登录鉴权 Token：用户登录后用于访问受保护接口
- Webhook 通知专属 Token：用于从固定通知调用路径定位并触发指定通知
- 渠道调用 Token：用于通过固定路径直接调用指定渠道

说明：
- 个人设置页不再保留用户级“调用 Token”能力

## 14. 当前 V1 边界总结

### 纳入 V1
- 多账号注册登录
- Token 鉴权
- 通知管理
- 渠道管理
- 推送记录
- 首页概览
- 测试模块
- 个人设置
- 多轮 AI 创建通知
- Webhook 通知专属 token 触发
- Webhook 模板（body JSON）

### 不纳入 V1
- 团队能力
- 邮件转发
- NAS 日志模块
- 通用模板系统
- 通知级立即发送一次
- Webhook query/header 模板变量
- 自动识别 webhook 字段并点选插入
