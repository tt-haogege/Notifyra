# Notifyra 执行清单

## 1. 适用范围

用于从“收到需求”到“开发完成”的标准执行路径，确保不同模型或新对话都能按统一节奏推进。

---

## 2. 开发前清单

开始前逐项确认：

- [ ] 已阅读相关设计文档
- [ ] 已确认当前任务所属阶段和任务编号
- [ ] 已确认当前任务边界（做什么 / 不做什么）
- [ ] 已阅读相关代码与测试
- [ ] 已确认是否需要先出计划
- [ ] 已确认是否需要 TDD
- [ ] 已确认是否需要 e2e
- [ ] 已确认是否涉及风险操作

---

## 3. 计划清单

如果任务需要先做计划，计划里至少要写清：

- [ ] Context
- [ ] 本阶段目标
- [ ] 不在本阶段做的内容
- [ ] 关键文件
- [ ] 必须复用的现有实现
- [ ] 实施步骤
- [ ] 验证命令

---

## 4. TDD 清单

- [ ] 先写失败测试
- [ ] 已实际运行并确认红灯
- [ ] 失败原因符合预期
- [ ] 写最小实现
- [ ] 再次运行测试确认绿灯
- [ ] 再做必要整理

---

## 5. 开发中清单

- [ ] 一次只推进一个明确目标
- [ ] 不提前做后续阶段能力
- [ ] 不顺手做无关重构
- [ ] 保持统一代码风格
- [ ] 每完成一块就做验证
- [ ] 用户隔离与边界逻辑未被破坏

---

## 6. Review 清单

### 简单任务
- [ ] 规格符合
- [ ] 测试通过
- [ ] 没有明显运行时问题
- [ ] 尽量一轮收口

### 中等及以上任务
- [ ] 先做规格 review
- [ ] 再做代码质量 review
- [ ] 问题集中时合并修复
- [ ] 避免低风险问题反复多轮拖长时间

---

## 7. 完成前清单

- [ ] 相关 unit tests 通过
- [ ] 相关 e2e tests 通过（如有）
- [ ] build 通过
- [ ] 关键行为符合设计文档
- [ ] 返回格式 / 状态码规范未被破坏
- [ ] 无当前范围内明显缺口

---

## 8. 文档清单

以下情况要同步更新文档：
- 开发流程变更
- review 流程变更
- 测试规范变更
- git / 提交流程变更
- 阶段边界或执行方式发生稳定变化
