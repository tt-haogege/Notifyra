# Notifyra 开发流程规范

## 1. 目的

本文档用于沉淀本项目的实际开发流程、协作约束与执行规范，减少重复沟通，统一后续开发节奏。

适用范围：
- backend
- frontend
- 设计文档补充
- 功能开发、重构、修复、联调

配套文档：
- `design/dev/README.md`
- `design/dev/ProjectOnboarding.md`
- `design/dev/ExecutionChecklist.md`
- `design/dev/BackendDevelopmentGuide.md`
- `design/dev/FrontendDevelopmentGuide.md`
- `design/dev/ReviewChecklist.md`
- `design/dev/TestingGuidelines.md`
- `design/dev/GitWorkflow.md`
- `design/dev/DocumentationMaintenance.md`

---

## 2. 总原则

### 2.1 先看设计，再开发
开始实现前，优先阅读并对齐以下文档：
- `design/RequirementDesign.md`
- `design/ProductDesign.md`
- `design/TechnicalDesign.md`
- `design/DevelopmentPlan.md`

如果已有相关专项文档，也要优先对齐，例如：
- `design/BackendStyleCleanupPlan.md`
- `design/dev/*.md`

### 2.2 用户确认优先
涉及以下情况，先给出方案或计划，再进入实现：
- 新功能
- 多文件改动
- 影响现有行为的修改
- 有多种可行实现路径的任务

### 2.3 尽量小步快跑
开发时遵循：
- 优先做最小闭环
- 先完成当前阶段目标，不提前扩展后续阶段能力
- 不顺手做无关重构
- 不为未来假设需求做过度抽象

### 2.4 简单任务不要拖长
对于范围明确、风险低、实现直接的任务：
- 减少不必要的 review 往返
- 尽量一次性收口
- review 要聚焦真实问题，不做形式化重复检查

## 3. 代码风格规范

### 3.1 基础风格
当前项目默认遵循：
- 使用分号
- 保持缩进和格式统一
- 尽量少用不必要的 `else`
- 优先使用 guard clause
- 不随意改动无关代码风格

### 3.2 修改前先参考周边代码
在改动已有文件前，应先阅读同目录已有实现，优先复用当前代码模式，避免风格漂移。

### 3.3 不做过度抽象
除非重复已经明显影响维护，否则不要提前抽出：
- 工具函数
- helper
- 基类
- 中间层封装

原则是：
- 当前任务够用即可
- 三五行重复通常优先保留
- 只有重复已经影响质量时再收敛

## 4. 测试驱动开发（TDD）规范

### 4.1 基本流程
默认遵循 TDD：
1. 先写失败测试
2. 运行测试，确认红灯且失败原因正确
3. 写最小实现
4. 再运行测试，确认绿灯
5. 再做必要整理

### 4.2 不允许跳过红灯确认
如果没有先看到测试失败，就不能认为测试有效。

### 4.3 测试范围
新增功能或行为变更时，至少补齐对应层级测试：
- service 单测
- controller / e2e 合约测试（按需要）
- build 验证

### 4.4 测试要覆盖真实边界
优先覆盖：
- 成功路径
- 用户隔离
- 参数校验
- 非法状态
- 外部依赖失败
- 返回格式稳定性

## 5. 文档与计划流程

### 5.1 什么时候要先做计划
以下任务优先先做计划：
- 新模块
- 跨多个文件的功能开发
- 有阶段边界约束的需求
- 需要和设计文档严格对齐的开发

### 5.2 计划文档内容
计划文档至少应包含：
- Context
- 目标边界
- 关键文件
- 复用点
- 实施步骤
- 验证步骤

### 5.3 流程文档放置位置
所有开发流程、工程规范、协作流程相关文档统一放在：
- `design/dev/`

### 5.4 流程变更后的维护要求
开发流程、review 流程、测试规范、git 规范、文档规则一旦发生稳定变化，必须同步更新 `design/dev/` 相关文档。

如果当前任务推进前已经明确需要补文档或更新流程文档，应先完成对应文档更新，再继续后续功能开发。

## 6. 实现流程规范

### 6.1 开发前
开始前应完成：
1. 阅读相关设计文档
2. 阅读相关代码
3. 判断是否需要先出计划
4. 明确当前阶段边界

### 6.2 开发中
开发中应遵循：
- 一次只推进一个明确目标
- 先完成当前任务闭环
- 每完成一块就做验证
- 不把简单任务做成过长流程

### 6.3 开发后
开发完成后至少应验证：
- 相关单测通过
- 相关 e2e 通过（如有）
- build 通过
- 统一响应格式未被破坏
- 关键行为与设计一致

---

## 7. Review 流程规范

### 7.1 Review 的目标
Review 的目标是找真实问题，不是机械重复。

重点检查：
- 是否符合设计边界
- 是否破坏现有行为
- 是否存在真实运行时风险
- 是否有明显多余复杂度
- 测试是否覆盖关键行为

### 7.2 Review 轮次控制
默认原则：
- 简单任务：尽量一轮收口
- 中等任务：实现后做规格 review + 质量 review
- 如果问题很集中，应合并修复，避免反复多轮拉长时间

### 7.3 什么情况必须继续 review
以下情况必须继续补 review：
- 存在真实运行时错误
- 规格未达成
- 测试没有覆盖关键路径
- build / test 未通过

### 7.4 什么情况不要过度 review
以下情况不要无限放大：
- 低风险的小重复
- 不影响当前范围的轻微抽象建议
- 纯风格争议但不影响一致性的点

## 8. 后端开发约定

### 8.1 backend 当前约定
当前 backend 开发应优先复用以下模式：
- NestJS module / controller / service 分层
- `PrismaService` 全局注入
- `JwtAuthGuard + CurrentUser` 用户隔离
- 全局响应格式：`{ code, data, message }`
- 全局异常过滤器统一输出

### 8.2 backend 测试约定
优先使用：
- service 层 unit test
- mocked PrismaService 的 e2e
- 不依赖真实 DB 的接口合约测试

### 8.3 渠道 / 通知这类核心模块
实现时要注意：
- 当前阶段只做当前阶段边界
- 不提前实现后续执行层、调度层、开放调用层
- 数据隔离必须始终按用户维度处理

---

## 9. 前端开发约定

### 9.1 前端实现优先级
前端优先保证：
- 结构正确
- 暗黑主题一致
- 间距与层级统一
- 交互闭环完整

### 9.2 与设计稿不一致时
如果设计稿和已有实现冲突：
- 优先回看设计文档
- 不确定时先给方案再修改

---

## 10. 提交与收尾规范

### 10.1 完成定义
一个开发项完成，至少表示：
- 代码已实现
- 相关测试已通过
- build 已通过
- 关键逻辑已 review
- 未遗留当前范围内的明显缺口

### 10.2 提交前检查
提交前至少确认：
- 不带入无关文件
- 不提交敏感信息
- 不包含未完成实验代码
- 提交说明能反映本次改动目的

### 10.3 不自动提交
除非用户明确要求，否则不要主动创建 git commit。
