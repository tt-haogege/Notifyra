# TechnicalDesignReview

## 项目名称

**Notifyra**

## 1. 评审结论

当前《TechnicalDesign.md》能够覆盖 Notifyra V1 的主要需求范围，整体分层、数据模型、核心流程与接口划分基本完整，可以作为开发拆解基础。

综合结论：**方案可实施，但建议在进入开发前补齐 6 个关键细节，以减少返工。**

---

## 2. 主要优点

1. **架构分层清晰**
   - API、调度器、Worker、驱动层职责明确
   - 适合通知执行类场景，便于后续扩展新渠道

2. **领域模型基本完整**
   - 已覆盖用户、通知、渠道、推送记录、AI 会话、Webhook 原始请求等核心对象
   - 能支撑当前 V1 页面与业务流程

3. **开放调用边界清楚**
   - 通知级 Webhook Token 与渠道级调用 Token 做了明确区分
   - 避免后期接口能力混淆

4. **执行记录设计较稳妥**
   - 标题、正文、渠道名都采用快照保留
   - 有利于追踪历史发送上下文

5. **已考虑关键安全问题**
   - token 哈希存储
   - AI key 加密
   - 数据按 user_id 隔离

---

## 3. 主要问题与优化建议

## 3.1 需求基线仍有历史冲突

### 问题
当前需求文档 `design/RequirementDesign.md` 仍存在两处历史内容未同步：
- 测试模块仍是旧版单入口描述
- 个人设置仍保留用户级“调用 Token”

### 风险
- 开发阶段如果同时参考 RequirementDesign 与 ProductDesign，容易出现实现口径不一致
- 接口、页面、测试用例都可能因此跑偏

### 建议
- 在进入开发前先补一轮需求文档同步
- 明确以以下结论为准：
  1. 测试模块是“测试通知 / 测试渠道”双入口
  2. 个人设置不包含用户级调用 Token

优先级：**高**

---

## 3.2 通知状态与展示状态还需要再细化

### 问题
技术方案中将一次性通知执行后定义为 `completed`，但产品侧页面是否仍允许继续查看、复制配置、二次编辑，没有完全说清。

### 风险
- 前端列表筛选与 switch 交互可能和已完成状态冲突
- 用户可能误解“已完成”和“已停用”的区别

### 建议
- 明确区分：
  - 运行态：active / disabled / blocked_no_channel
  - 生命周期态：completed（仅一次性通知执行后）
- 前端展示上建议：
  - 已完成的一次性通知不展示启停 switch
  - 允许查看详情和复制创建新通知

优先级：**中**

---

## 3.3 测试模块“是否入库”需要统一口径

### 问题
技术方案提到测试结果不混入正式统计，但也提到“可视情况保留审计日志”，目前还不够明确。

### 风险
- 如果完全不落库，测试历史无法追踪
- 如果复用 push_records，又容易污染概览统计和正式记录筛选

### 建议
两种可选方案中，推荐方案 A：

#### 方案 A（推荐）
- 测试结果仍写入 `push_records`
- 但 `source` 标记为 `test_notification` / `test_channel`
- 概览统计、正式列表默认排除测试 source

**优点：**
- 复用现有模型
- 便于追踪问题
- 后端实现更简单

#### 方案 B
- 单独建设 test_records 表

**缺点：**
- V1 复杂度增加
- 后续维护成本更高

优先级：**高**

---

## 3.4 开放 token 的展示与重置交互还需要补前端约束

### 问题
技术方案中明确 token 仅生成时明文返回一次，但没有把前端交互规则写死。

### 风险
- 前端若默认可反查完整 token，会与“仅存哈希”矛盾
- 用户可能在未复制时关闭弹窗，后续误以为还能再次查看完整值

### 建议
补充产品/技术统一约束：
- 新建或重置 token 后，仅在当前弹窗中展示一次完整值
- 关闭后只显示脱敏值
- UI 必须提示“请立即保存，关闭后无法再次查看完整 Token”

优先级：**高**

---

## 3.5 Webhook 模板路径建议加白名单解析约束

### 问题
当前模板规则只说明支持 `{{body.xxx}}`，但没有定义最大嵌套层级、非法路径处理方式、数组访问规则。

### 风险
- 模板解析器实现口径不一
- 容易出现边界行为不一致

### 建议
补充以下约束：
- 仅允许 `body` 根路径
- 支持点路径，不支持表达式
- 暂不支持数组索引，如 `body.items[0]`
- 单次内容最多解析固定数量占位符，如 50 个
- 非法路径原样保留，不报错中断发送

优先级：**中**

---

## 3.6 调度实现需明确时区口径

### 问题
当前技术方案提到 Cron 与 next_trigger_at，但没有明确时区来源。

### 风险
- 用户在“下午 / 明早”偏好、Cron 触发、概览统计时可能出现时区偏差
- 本地开发与生产环境容易行为不同

### 建议
- V1 固定采用“用户个人设置时区”或“系统统一时区”二选一
- 若 V1 不做用户时区配置，建议先统一使用服务端固定时区，并在文档中写明
- 所有 next_trigger_at 都转换为 UTC 存储，展示时再转本地时区

优先级：**高**

---

## 4. 可进一步优化但不阻塞开发的点

## 4.1 渠道配置 schema 可再标准化
建议为不同渠道建立独立配置校验 schema，避免 `config_json` 变成无约束大对象。

## 4.2 示例代码生成可模板化
建议代码示例采用模板引擎生成，而不是在接口里手写字符串拼接，便于后续扩语言。

## 4.3 概览统计可做轻缓存
若后续推送量增加，概览统计建议做短时缓存，如 30~60 秒。

## 4.4 删除策略建议优先逻辑删除
通知与渠道若后续需要审计回溯，建议优先逻辑删除而非物理删除。

---

## 5. 建议的开发前补充清单

进入开发前建议先补齐以下事项：

1. 同步更新 `design/RequirementDesign.md`
2. 明确测试结果是否进入 `push_records`，推荐进入并用 source 区分
3. 明确 token 一次性展示规则
4. 明确 Cron / next_trigger_at 的时区口径
5. 明确一次性通知执行后的前端展示规则
6. 明确 Webhook 模板路径的白名单语法
7. 将 SQLite、本地数据目录 `/data/`、以及 `node-schedule` 调度方案固定为当前 V1 技术基线

---

## 6. 最终建议

技术方案整体可继续推进开发，但建议先完成一轮“小范围设计收口”，优先解决：
- 需求文档同步
- 测试记录入库口径
- token 展示规则
- 时区口径

这四项如果先定下来，后续接口定义、前端交互和测试用例都会更稳。
