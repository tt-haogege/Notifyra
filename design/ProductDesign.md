# ProductDesign

## 项目名称

**Notifyra**

## 1. 文档目的

本文档基于已确认的需求，对 Notifyra V1 的产品结构、页面设计、交互原则和核心页面方案进行整理，用于后续原型深化、技术设计与开发实现。

---

## 2. 产品定位

Notifyra 是一个面向个人与多账号独立用户的聚合通知平台。

V1 聚焦于：
- 聚合多种通知渠道
- 支持手动创建与 AI 多轮创建通知
- 支持一次性、周期性、Webhook 三类通知触发方式
- 提供通知管理、渠道管理、推送记录、测试模块与个人设置
- 提供通知级 Webhook Token 和渠道级调用 Token 能力

---

## 3. 全局设计原则

### 3.1 产品风格方向
本轮已确认的视觉方向为：
- 深色玻璃感左侧导航
- 浅色高质感主内容区
- 蓝紫系柔光高亮
- 整体偏高级、稳重、长期耐看
- 不走强赛博霓虹路线

### 3.2 结构原则
全站遵循：
- 配置在前
- 执行在中
- 记录在后

即：
- 通知 / 渠道 / 设置负责配置
- Webhook / 渠道调用 / 定时负责执行
- 推送记录 / 概览负责反馈

### 3.3 交互原则
- 页面信息分区要清楚
- 高频操作优先显性展示
- 低频说明放提示区或详情区
- 状态操作优先使用 switch
- 详情 / 编辑 / 删除等动作尽量图标化
- Webhook 专属信息只在 Webhook 类型上下文里出现

---

## 4. 一级模块结构

Notifyra V1 主导航包含：

1. **概览**
2. **通知**
3. **渠道**
4. **推送记录**
5. **测试模块**
6. **个人设置**

### 4.1 左侧导航规则
左侧导航采用深色玻璃感菜单。

导航底部放置：
- 用户头像
- 用户名
- 主题切换
- 退出登录

该结构全站统一，不在页面顶部重复放置用户信息。

---

## 5. 页面流转主线

### 5.1 首次使用主路径
注册 / 登录 → 概览页 → 添加渠道 → 新建通知 → 查看推送记录

### 5.2 日常使用路径
概览页 → 通知列表 → 通知详情 / 编辑 → 推送记录

### 5.3 AI 创建路径
新建通知 → 多轮 AI 对话 → 创建成功 → 跳转通知详情

### 5.4 外部调用路径
#### 通知级调用
通知详情 → 查看 Webhook Token → 外部系统调用 → 推送记录

#### 渠道级调用
渠道详情 → 查看渠道调用 Token → 外部系统调用 → 推送记录

### 5.5 测试路径
测试模块 → 选择测试对象（通知 / 渠道）→ 发送测试 → 查看结果 → 查看示例代码

## 6. 核心页面方案

## 6.1 概览页

### 页面作用
- 展示系统整体状态
- 提供高频入口
- 展示最近动态

### 页面结构
1. 页面标题区
2. 核心统计卡片区
3. 快捷操作区
4. 最近推送记录区

### 展示内容
- 通知数量
- 成功数量
- 失败数量
- 最近的推送记录

### 快捷入口
1. 添加渠道
2. 新建通知
3. 测试模块
4. 推送记录

### 设计结论
- 快捷入口不使用普通按钮
- 使用更具产品感的操作面板样式
- 统计卡片做主次分层，不使用普通平铺白卡

---

## 6.2 通知列表页

### 页面作用
- 查看所有通知
- 快速判断状态
- 快速进入详情、编辑、启停操作

### 页面结构
1. 页面标题区
2. 筛选 / 搜索区
3. 通知列表区

### 顶部操作
- AI 创建
- 新建通知

### 筛选项
- 搜索通知名称
- 触发类型筛选
- 状态筛选

### 列表字段
- 通知名称
- 通知触发类型
- 渠道信息
- 状态
- 下一次触发时间
- 最近一次推送结果

### 渠道展示规则
- 展示：用户自定义渠道名 + 渠道类型
- 多渠道时最多展示前 2 个，其余折叠

### 行内操作
- 启用 / 停用：使用 switch
- 详情 / 编辑 / 删除：使用图标

---

## 6.3 新建通知页

### 页面作用
同时承载：
- AI 创建通知
- 手动创建通知

### 页面结构
1. 顶部 AI 创建区
2. 下方手动创建表单区

### AI 创建区
- 放在页面顶部
- 支持一句话发起
- 实际进入多轮 AI 对话

### 手动创建区字段
- 通知名称
- 发送渠道
- 标题
- 正文
- 触发类型
- 触发时间（动态显示）

### 触发类型交互规则
支持三类：
1. 一次性通知
2. 周期性通知
3. Webhook 通知

### 触发时间字段规则
- 字段名统一为：**触发时间**
- 当触发类型为 **一次性通知** 时：
  - 显示日期时间选择器
- 当触发类型为 **周期性通知** 时：
  - 显示输入框
  - 提示输入 Cron 表达式
- 当触发类型为 **Webhook 通知** 时：
  - 不展示“触发时间”字段
  - 后续展示 Webhook 专属配置区域

---

## 6.4 通知详情页

### 页面作用
展示一条通知的完整信息与最近执行情况。

### 页面结构
左侧主信息区：
1. 基础信息
2. 内容信息
3. 最近推送记录

右侧辅助信息区：
1. 渠道信息
2. Webhook 信息（条件展示）
3. 说明区

### 顶部信息
- 通知名称
- 通知触发类型
- 当前状态
- 下一次触发时间 / 按请求触发
- 编辑 / 删除 / 启停操作

### Webhook 展示规则
- **只有当通知触发类型为 Webhook 时，才展示 Webhook 信息区**
- 其他类型通知不展示该区块

### Webhook 信息区内容
- 固定调用路径
- 调用 Token（脱敏显示，可复制）
- 查看示例
- 重置 Token

### 页面作用
- 查看渠道完整配置

- 查看推送执行历史
- 快速筛选与定位问题

### 页面结构
1. 页面标题区
2. 顶部推送统计区
3. 筛选区

## 6.8 推送记录详情页

### 页面作用
- 展示一次具体推送的完整上下文
- 查看失败原因
- 查看各渠道执行结果
- 如为 Webhook 触发，则查看原始请求内容

### 页面结构
左侧：
1. 基础信息
2. 发送内容
3. Webhook 原始请求（条件展示）

右侧：
1. 渠道结果
2. 失败说明
3. 快捷跳转

### 渠道结果
逐条展示：
- 渠道名称
- 成功 / 失败状态
- 失败原因

### Webhook 原始请求区内容
- 请求时间
- 来源 IP
- 请求体 Body

### 展示规则
- 仅在该条记录由 Webhook 触发时展示原始请求信息

## 6.9 测试模块页

### 页面作用
- 验证通知或渠道是否工作正常
- 生成对应接入示例代码

### 页面结构
测试模块固定采用顶部双入口结构：
1. **测试通知**
2. **测试渠道**

建议使用顶部 Tab 切换。

### 入口一：测试通知
#### 页面目标
- 验证一条已存在通知在当前配置下是否可以正常发送

#### 交互方式
- 先选择一条已有通知
- 系统读取该通知当前配置
- 执行一次测试发送
- 展示测试结果
- 同步生成对应示例代码

#### 说明
- 该能力用于验证“这条通知当前配置是否可工作”
- 这里的测试行为不等同于正式通知执行记录，应以测试结果视图承载

### 入口二：测试渠道
#### 页面目标
- 验证一个已存在渠道是否可用

#### 交互方式
- 先选择一个已有渠道
- 针对该渠道发送测试消息
- 展示测试结果
- 同步生成该渠道对应的接入示例代码

#### 说明
- 该能力用于单独验证渠道连通性和发送能力
- 不依赖某条通知存在

### 页面布局建议
在任一入口下，页面均采用左右分区：

左侧：
1. 测试对象选择区
2. 测试发送区
3. 测试结果区

右侧：
1. 示例代码区
2. 提示说明区
3. 相关快捷入口

### 示例代码规则
- 示例代码根据当前测试入口联动变化
- 支持：curl / JavaScript / Python
- 当测试对象为通知时，代码偏向通知调用方式
- 当测试对象为渠道时，代码偏向渠道调用方式

### 补充规则
- 测试模块不再采用“直接输入一段测试内容然后随意发送”的单入口方式
- 必须先明确测试对象是“通知”还是“渠道”
- 测试消息不等同于执行已有通知一次
- 测试模块是独立验证与接入辅助工具

## 7. 本轮已确认的关键 UI 规则

1. 左侧菜单采用**高级玻璃感**风格，不用赛博霓虹重风格
2. 菜单底部放置头像、用户名、主题切换、退出登录
3. 通知列表中，启用 / 停用使用 switch
4. 列表中的详情 / 编辑 / 删除使用图标
5. Webhook 信息只在 Webhook 类型上下文里展示
6. 渠道页 warning 提示放在搜索框上方，使用黄色提示条
7. 推送记录统计放在列表上方
8. 测试模块使用顶部 Tab 双入口：测试通知 / 测试渠道
9. 测试模块必须先选择测试对象，再进行发送与代码生成
10. 个人设置页不再保留“调用 Token”能力
